## 1. 引言

### 1.1 目的
本文档定义门把手触摸功能的**软件集成测试策略**，明确测试范围、方法、环境、通过准则及职责分工，确保各软件模块在集成后满足 SRS 中的功能与非功能需求。

### 1.2 范围
- 覆盖以下模块的集成行为：
  - `Cap_Wrapper`（厂商 SDK 接口）
  - `Signal_Controller`（开锁信号输出）
  - `Power_Manager`（运行/睡眠切换）
  - `LIN_Diag`（诊断服务）
  - `OTA_Handler`（固件更新）
- **不包含**：
  - 厂商 SDK 内部逻辑（视为黑盒）
  - 硬件电路可靠性测试（如 ESD、温漂）
  - HIL 测试（项目已取消）

### 1.3 参考文档
- 《软件需求分析说明书》SRS-DOORHANDLE-TOUCH-V1.5
- 《软件详细设计文档》SDD-DOORHANDLE-TOUCH-V1.0
- 项目测试管理规范 V1.0

## 2. 测试目标

| 目标类别 | 具体目标 |
|--------|--------|
| **功能集成** | 验证触摸事件 → 开锁信号 → 电源管理的端到端流程 |
| **通信集成** | 验证 LIN 诊断与 OTA 在系统运行/睡眠下的正确性 |
| **状态一致性** | 验证各模块在运行、睡眠、唤醒状态下的协同行为 |
| **错误处理** | 验证异常输入（如无效 LIN 帧、SDK 初始化失败）下的系统鲁棒性 |

---

## 3. 测试级别与类型

### 3.1 集成测试级别
采用**自底向上 + 功能场景驱动**结合策略：

| 阶段 | 测试内容 | 方法 |
|------|--------|------|
| **阶段1：核心功能集成** | `Cap_Wrapper` + `Signal_Controller` + `Power_Manager` | 台架测试 + 示波器观测 |
| **阶段2：通信集成** | + `LIN_Diag` + `OTA_Handler` | LIN 仿真 + 固件更新验证 |
| **阶段3：全系统集成** | 所有模块 + 电源管理 + NVM | 场景化压力测试 |

### 3.2 测试类型
- **功能测试**：验证需求覆盖（SRS ID → 测试用例）
- **时序测试**：验证响应延迟（如触摸→信号≤20ms）
- **状态切换测试**：运行 ↔ 睡眠 ↔ 唤醒
- **边界/异常测试**：无效触摸、LIN 错误帧、OTA 中断
- **NVM 持久性测试**：断电后数据恢复验证

## 6. 通过/失败准则

### 6.1 通过准则
- 所有 **SRS 需求**均有对应测试用例且通过；

- SIG满足：

   	输出信号电平：低电平且≤0.6V;

  ​	 输出信号持续时间： 330ms~2000ms （±20）;

- 睡眠电流 ≤ 200μA；

- LIN 诊断响应时间 ≤ 10ms；

### 6.2 失败处理
- 任一**关键功能**（如开锁信号、睡眠唤醒）失败 → **阻塞发布**；
- 非关键问题（如诊断 DID 字段格式错误） → 记录为低优先级缺陷，可带缺陷发布（需评审）。

## 8. 风险与缓解

| 风险 | 缓解措施 |
|------|--------|
| 厂商 SDK 行为与文档不符 | 要求供应商提供测试报告；增加黑盒验证用例 |
| LIN 仿真工具不支持 OTA 流程 | 开发简易 Python 脚本模拟 0x34/0x36 流程 |
| 睡眠电流测量不准确 | 使用高精度电流分析仪（如 Keysight N6705C） |
| NVM 写入寿命不足 | 限制写入频率（每 10 次触摸写一次） |
