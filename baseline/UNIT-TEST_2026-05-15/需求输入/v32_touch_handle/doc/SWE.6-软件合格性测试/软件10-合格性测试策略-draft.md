## 1. 引言

### 1.1 目的
本文档定义门把手触摸功能的**软件合格性测试策略**，旨在通过系统化的黑盒测试验证软件是否满足《软件需求分析说明书》（SRS）中定义的所有需求，确保软件在交付前具备功能正确性、稳定性与可接受的质量水平。

> **注**：本项目为**非功能安全（Non-ASIL）舒适性功能**，合格性测试聚焦于需求覆盖、用户场景与边界行为，**不包含 ISO 26262 要求的软件单元/集成验证**。

### 1.2 范围
- **测试对象**：门把手触摸功能完整软件（含 Bootloader、Application、LIN 诊断、OTA 逻辑）
- **覆盖内容**：
  - 所有 SRS 功能与非功能需求（V1.5）
  - 用户典型使用场景
  - 异常输入与恢复能力
  - 固件升级与诊断接口
- **不包含**：
  - 硬件可靠性测试（如寿命、EMC）
  - 厂商 SDK 内部逻辑验证
  - HIL 或实车测试（由系统级测试覆盖）

### 1.3 参考文档
- 《软件需求分析说明书》SRS-DOORHANDLE-TOUCH-V1.5
- 《软件集成测试用例》STC-DOORHANDLE-INT-V1.0
- 项目软件发布 checklist V1.0

## 2. 测试目标

| 目标类别 | 具体目标 |
|--------|--------|
| **需求符合性** | 100% 覆盖 SRS 中定义的 45 项需求（含功能、性能、接口、可测试性） |
| **场景完整性** | 覆盖用户正常使用、误操作、环境干扰等典型场景 |
| **鲁棒性验证** | 验证软件在异常输入、通信错误、电源波动下的行为 |
| **可交付性** | 确保软件满足发布基线要求（无高优先级缺陷） |

---

## 3. 测试方法与级别

### 3.1 测试方法
采用**黑盒测试（Black-Box Testing）** 为主，结合**场景驱动**与**需求追踪**：

| 方法 | 说明 |
|------|------|
| **需求追踪测试** | 每项 SRS 需求映射至少 1 个合格性测试用例 |
| **场景测试** | 模拟真实用户操作流程（如“触摸 → 解锁 → 睡眠”） |
| **边界值分析** | 测试输入边界（如脉宽 450ms / 550ms） |
| **错误注入** | 人为制造 LIN 错帧、断电、传感器异常等 |

### 3.2 测试级别
合格性测试为**系统级黑盒测试**，在**完整软硬件原型上执行**，不区分模块。

## 5. 测试内容分类

### 5.1 功能合格性
验证软件是否按需求执行核心功能：
- 触摸检测 → 开锁信号输出（500ms 高电平）
- 单次触发防重复
- 自动睡眠与唤醒

### 5.2 性能合格性
- 响应延迟 ≤ 20ms（触摸 → 信号）
- 睡眠电流 ≤ 50μA
- OTA 更新时间 ≤ 90 秒（128KB）

### 5.3 接口合格性
- LIN 诊断：支持 DID 0xF180 / 0xF190 读取
- OTA：支持标准 0x34/0x36 流程
- GPIO 电平：兼容 3.3V/5V 逻辑

### 5.4 鲁棒性与异常处理
- 无效 LIN 帧：返回正确 NRC
- OTA CRC 错误：拒绝激活
- 断电后：NVM 数据不丢失
- 传感器失效：降级运行并上报错误

### 5.5 环境适应性
- -40°C / +85°C 下功能正常
- 电源电压 9V–16V 范围内稳定工作

## 6. 通过/失败准则

### 6.1 通过准则（必须全部满足）
- **100% SRS 需求覆盖**：所有 45 项需求均有对应测试用例并通过；
- **关键功能零缺陷**：
  - 开锁信号输出正确
  - 睡眠/唤醒正常
  - LIN 诊断可读
- **性能达标**：时序、电流、OTA 时间符合要求；
- **无高优先级缺陷**：无阻塞用户核心功能的问题。

### 6.2 缺陷分类与处理
| 优先级 | 定义 | 处理要求 |
|--------|------|--------|
| **高（Critical）** | 导致开锁失败、无法唤醒、OTA 永久失效 | 必须修复，否则拒绝发布 |
| **中（Major）** | 诊断读错、计数偏差、LED 异常 | 需评审，可带缺陷发布（记录偏差） |
| **低（Minor）** | 日志格式错误、非关键时序轻微超差 | 可接受，记录于已知问题列表 |

## 7. 测试执行计划

| 阶段 | 活动 | 周期 | 交付物 |
|------|------|------|--------|
| **准备** | 搭建环境、准备固件、评审用例 | 2 天 | 测试环境就绪报告 |
| **执行** | 执行全部合格性用例 | 5 天 | 测试执行记录、缺陷报告 |
| **回归** | 修复后回归关键路径 | 1–2 天 | 回归测试报告 |
| **评审** | 发布前质量评审 | 1 天 | 合格性测试总结报告 |

> **总周期**：约 **7–8 个工作日**

## 9. 风险与缓解

| 风险 | 缓解措施 |
|------|--------|
| 原型硬件供应不足 | 提前锁定 3 套测试样件，禁止挪用 |
| LIN 工具不支持 OTA | 开发轻量级 Python LIN 脚本替代 |
| 环境测试资源冲突 | 提前预约高低温箱，采用分时测试 |
| 需求理解偏差 | 测试用例需经需求作者（SRS 编写人）评审 |
