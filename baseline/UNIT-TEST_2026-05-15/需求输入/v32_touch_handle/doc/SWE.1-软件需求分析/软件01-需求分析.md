## 1. 引言

### 1.1 目的
本文档定义用于汽车门把手触摸感应功能的软件需求。根据最新硬件与驱动策略变更：**电容触摸检测由芯片厂商提供的软件库（SDK 或 BSP）完成**，本软件调用其标准接口获取“触摸事件”结果，**根据电容信号变化、滤波或阈值判定，再结合软件算法判断是否需要触发**。输出最短320ms，最长2s的低电平开锁信号，通信保留 LIN 用于诊断、 OTA与反馈把手状态。

### 1.2 范围
- 调用芯片厂商提供的触摸检测接口（如 `Cap_GetTouchEvent()`）；
- 响应有效触摸事件，输出320ms~2s开锁信号；
- 管理系统低功耗状态；
- 支持 LIN 诊断(OTA、DID) ；

### 1.3 定义与缩写
- **厂商 SDK**：电容传感芯片供应商提供的软件开发包，含初始化、中断处理、事件查询等 API；
- **触摸事件**：由厂商 SDK 封装后的布尔型或枚举型，再通过软件算法判断输出（如 TOUCHED / NOT_TOUCHED）；
- **LIN**：Local Interconnect Network；
- **OTA**：Over-The-Air 固件更新；
- **GPIO**：通用输入输出。

## 2. 总体描述

### 2.1 产品概述
本软件运行于门把手内部 MCU，集成芯片厂商提供的**电容触摸检测 SDK**。系统初始化后，定期或通过中断调用 SDK 接口获取“是否发生有效触摸”的结果。一旦确认触摸事件，软件立即输出一个**最短320ms，最长2s的 低电平开锁信号**（`UNLOCK_SIGNAL`），供车身控制器使用。所有底层电容信号处理（包括滤波、基线校准、防抖、抗干扰等）均由厂商 SDK 实现。

### 2.2 使用场景
- 用户触摸门把手 → 厂商 SDK 检测并标记有效触摸 → 本软件调用接口获知事件 → 输出 低电平开锁信号；
- 无操作 5 秒后，系统进入睡眠；
- 售后通过 LIN 读取状态或升级固件。

### 2.3 产品变体
**无变体**。全系功能一致。

### 2.4 电源管理
- **运行功耗**：<15mA（含 MCU 与传感器 IC）；
- **睡眠功耗**：<200μA；
- **唤醒源**：厂商 SDK 支持的中断引脚（`CAP_INT`）或 LIN 总线活动。

## 3. 外部接口

### 3.1 硬件接口
| 接口 | 描述 |
|------|------|
| **电容传感器模块** | 集成厂商 SDK 的传感 IC（如 Cypress CAPSENSE™、Microchip MTCH 系列等） |
| **开锁信号输出** | `UNLOCK_SIGNAL`：GPIO，低电平有效 |
| **中断引脚** | `CAP_INT`：连接至 MCU，由 SDK 配置为触摸事件中断 |
| **LIN 接口** | 单线通信，19200 bps |
| **电源** | 12V 输入，内部 LDO |

### 3.2 软件接口（与厂商 SDK 交互）
- **初始化**：`Cap_Init()` — 配置传感器、中断、扫描参数；
- **事件查询**：`Cap_GetTouchEvent()` → 返回 `bool`（true = 有效触摸）；
- **中断回调**（可选）：`Cap_InterruptHandler()` — 由 SDK 触发；
- **低功耗控制**：`Cap_EnterSleep()` / `Cap_ExitSleep()`。

> **注**：具体 API 名称以所选芯片厂商 SDK 为准，本需求仅定义行为接口。

### 3.3 输出信号
- **`UNLOCK_SIGNAL`**：
  - 电平：低电平有效（＜0.8V）；
  - 脉宽：320ms ±20~2000ms±20；
  - 默认状态：高电平。

## 4. 功能需求

### 4.1 触摸事件获取
- **ID: 2063**
  系统启动时调用 `touch_init()` 完成传感器初始化；
- **ID: 2064**
  通过轮询(DEF_TOUCH_MEASUREMENT_PERIOD_MS)和中断方式调用 `touch_sensors_config` 获取触摸状态；

### 4.2 开锁信号输出
- **ID: 1685**
  当 `measurement_done_touch == true` 时，立即置 `UNLOCK_SIGNAL = LOW`；
- **ID: 2067**
  启动 1ms 定时器，触发后开始计数并持续判断measurement_done_touch ，当计数达到320后measurement_done_touch 依旧为true,则继续计数直到计数达到2000或者measurement_done_touch == false时拉低 `UNLOCK_SIGNAL`；

### 4.3 电源与模式管理
- **ID: 2064**（运行模式）
  系统持续轮询或监听中断，处理触摸事件；
- **ID: 1739**（睡眠模式）
  若连续 5 秒无触摸事件，调用 `touch_enable_lowpower_measurement()` 并使 MCU 进入低功耗模式；
- **ID: 2065**（唤醒）
  任一唤醒源（`TOUCH` 或 LIN）激活后，调用 `touch_disable_lowpower_measurement()` 恢复传感器工作。

## 5. 非功能需求

### 5.1 性能
- **ID: 1705**
  从厂商 SDK 报告触摸事件到 `UNLOCK_SIGNAL` 下降沿延迟 ≤ 100ms；

### 5.2 可靠性
- **ID: 1864**
  厂商 SDK 需支持动态基线校准、抗水、抗电磁干扰；
- **ID: 1865**
  工作温度：-40°C 至 +85°C；
- **ID: 1860**
  支持 ≥ 200,000 次有效触发。

### 5.3 可测试性
- **ID: 2066**
  可通过物理触摸或厂商提供的仿真工具验证事件上报；
- **ID: 2067**
  示波器验证 `UNLOCK_SIGNAL` 脉宽（320ms ±20~2000ms±20）；
- **ID: 2068**
  LIN 诊断支持读取：软件版本、SDK 版本，触发状态。

## 6. 软件架构

### 6.1 模块划分
- `touch_init`：封装厂商 SDK 调用（初始化、事件获取、睡眠控制）；
- `touch_signal_input_task`：管理 `UNLOCK_SIGNAL` 输出时序与互斥逻辑；
- `low_power_task`：协调系统与传感器睡眠/唤醒；
- `lin_task`：实现lin数据接收，向uds传递数据；
- `analyze_lin_frame_v3`：解析lin数据，处理需要响应的lin数据。

### 6.2 关键依赖
- 厂商 SDK 必须提供：
  - 稳定的触摸事件输出接口；
  - 低功耗睡眠/唤醒支持；
  - 中断或轮询访问方式；
  - 通信功能的正常接收与响应；
  - 抗环境干扰能力（水、噪声、温漂）。

> **说明**：本软件**不实现任何电容算法**，仅作为“事件消费者”使用 SDK 输出。

## 9. 问题和变更

### 9.1 本次变更说明
- **抽象电容处理**：将底层触摸检测完全委托给芯片厂商 SDK；
- **降低开发风险**：避免自研滤波/防抖算法带来的调试复杂度；
- **提升可靠性**：依赖成熟商业 SDK 的抗干扰与环境适应能力；
- **简化本软件职责**：仅关注事件响应与信号输出。

### 9.2 已关闭问题
- **问题单 **: “自研电容算法在潮湿环境下误触发率高。”
  → **解决**：切换至厂商 SDK，其已内置水膜抑制与动态校准。
- **问题单 **: “团队缺乏电容信号调试经验。”
  → **解决**：采用黑盒 SDK 方案，仅需集成标准接口。
