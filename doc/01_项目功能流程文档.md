# V32 车规级电容式外把手 — 项目功能流程文档

> **芯片平台**: App 侧 — TCAE10 (ARM Cortex-M0+); Boot 侧 — TCPL03X (ARM Cortex-M0+)
> **通信总线**: LIN 2.x / J2602 (Slave 模式，波特率 19200bps)
> **诊断协议**: UDS on LIN (ISO 14229-1 / LIN 诊断规范)
> **调度内核**: App 侧 — LiteTask (轻量合作式调度器); Boot 侧 — 裸机超级循环
> **OTA 机制**: Bootloader + App 双区架构，通过 UDS 诊断服务实现固件下载与激活
> **版本**: V2.0 (基于编译源文件修正)
> **日期**: 2026-05-27

### 1.1 内存布局 (Flash 映射)

| 区域 | 起始地址 | 大小 | 内容 |
|------|---------|------|------|
| Bootloader 入口 | `0x00000000` | — | Boot 向量表 + 启动代码 |
| Bootloader 代码 | Boot 入口后 | \~7.5KB (V3) | Boot 固件 (TCPL03X) |
| 用户数据头 | Boot 后 (FLASH_APP_ADDR) | 0x100 (256B) | App 参数/NAD 配置头 |
| App 应用程序 | 用户数据头 + 0x100 | ~54KB | 应用程序固件 (TCAE10) |
| DFU 信息区 | Flash 指定扇区 | 512B (1 Sector) | 上次升级记录 (magic=0xDEADBEEF) |

> **注**: `JumpToApp()` 跳过 App 起始的 0x100 字节用户数据头，从 `FLASH_APP_ADDR + 0x100` 提取 MSP 和入口地址。

### 2.1 启动序列

```

      ▼

      │       │

      │       │     ├── 时钟配置 (CRG, 48MHz)
      │       │     ├── 中断向量表重定位

      │       │     ├── GPIO / PWM / ADC 初始化

      │       │
```

### 2.2 Bootloader 启动决策逻辑

```
上电复位 → Bootloader 初始化 → while(1) { main_loops() }
                                      │
               ┌──────────────────────┴──────────────────────┐
               │                                             │
        每循环: lin_diag_service_handle()             delay_ms(1) + LoopCnt++
               │                                       当 LoopCnt > 50
               │                                       (约 50ms)
               │                                             │
               │                                     ┌───────┴───────┐
               │                                     │               │
               │                              DFU 有效?        DFU 无效?
               │                                     │               │
               │                              BOOT_STATE_    BOOT_STATE_
               │                              USER_APP       UPGRADE
               │                                     │               │
               │                              JumpToApp()    等待 OTA 指令
               │
        如果在 IDLE 期间收到 $10 $02 (编程会话)
              → boot_state = BOOT_STATE_UPGRADE
              → 进入 DFU 状态机
```

**关键设计点**:
- Bootloader 上电后先等待约 50ms (50 × ~1ms) 监听诊断请求，确保可以立即响应 OTA 指令
- 在这 50ms 窗口内收到 `$10 $02` (编程会话) 则进入升级模式
- 超时后持续监听但不自动跳转 (仅在 DFU 信息有效时跳转 App)
- `sysTick_Handler` → `os_task_update()` 被 `dfu_uds_manager.c` 覆盖，仅做超时管理，不使用 os_manager 任务调度器

## 3. App 任务调度 (LiteTask RTOS)

### 3.1 LiteTask 内核概述

App 侧使用 **LiteTask** (`app/os/litetask/`)，一个轻量、非抢占、合作式消息调度器，非完整 RTOS。

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大任务数 | 5 (TC_TASK_NUM) | 固定池，无动态分配 |
| 消息队列 | 3/任务 (TC_MSG_NUM) | 固定大小槽位 |
| 定时器 | 5 (TC_TIMER_NUM) | CIRCLE(循环) / ONESHOT(单次) |
| SysTick | 1000 Hz | tc_port.c 配置 |
| 调度模式 | 非抢占轮询 | 任务运行至完成 |

**关键源文件**:

| 文件 | 作用 |
|------|------|
| `app/os/litetask/tc.c` | 内核: `TcTaskCreate()`, `TcTaskExec()`, `TcTimerCreate()`, `TcTimerExec()` |
| `app/os/litetask/tc_port.c` | 移植层: `SysTick_Handler()`, `TcPortInit()` |
| `app/os/litetask/tc.h` | 类型定义: `T_TcTask`, `T_TcTimer`, 消息 ID |
| `app/projects/.../Source/tc_conf.h` | 配置: TC_TASK_NUM=5, TC_TIMER_NUM=5 |

### 3.2 任务表

| 任务名 | 句柄 | 优先级 | 定时器周期 | 创建于 | 主要职责 |
|--------|------|--------|-----------|--------|---------|
| **HaltTask** | `HaltTask` | CRITICAL | `ENTER_HALT_TIMEOUT_MS` | `tc_halt.c` | 低功耗状态机: 休眠入口/唤醒过滤/Sleepwalk |
| **TouchTask** | `TouchTask` | HIGH | **1 ms** | `si_touch_port.c` | 电容触摸扫描 (快慢扫切换)，触摸算法执行 |
| **LinTask** | `LinTask` | HIGH | **1 ms** | `lin_task.c` | LIN 诊断服务处理, 未配置帧处理, SNPD, 低功耗管理 |
| **AppTask** | `AppTask` | MID | **2 ms** | `app.c` | LIN 控制报文, 会话检查, Flash 例程 27, 门驱动 PWM |

### 3.3 任务回调模式

所有任务遵循统一的消息驱动回调模式:

```c
static void TaskX(uint32_t msg, void *param)
{
    if (msg == MSG_TASK_INIT)      // 0: 任务创建时自动收到
    {
        // ● 创建循环定时器
        timer = TcTimerCreate(TC_TIMER_TYPE_CIRCLE, periodMs, NULL,
                              currentTask, NULL);
        TcTimerStart(timer);
        // ● 注册休眠过滤/监控回调
        HaltFilterWakeupRegister(FilterCb);
        HaltMonitorRegister(MonitorCb);
    }
    if (msg == MSG_TASK_TIMER)     // 1: 定时器到期时触发
    {
        do_periodic_work();        // ← 周期性核心逻辑在此执行
    }
    if (msg == MSG_TASK_ENTER_HALT || msg == MSG_TASK_WAKE_UP)
    {
        /* 低功耗生命周期钩子 */
    }
}
```

### 3.4 LinTask 详细行为 (1ms 定时器)

```
LinTask (每 1ms)
  ├── lin_diag_service_handle()   ── UDS 诊断服务分发
  ├── lin_uncd_frame_handle()     ── 未配置 LIN 帧处理
  ├── lin_lld_sci_timeout()       ── SCI 超时管理
  ├── lin_snpd_process_handle()   ── SNPD 自动寻址状态机
  ├── 低功耗超时管理
  ├── Slave 唤醒检测 (1s 周期)
  └── Break 计数监控
```

### 3.5 AppTask 详细行为 (2ms 定时器)

```
AppTask (每 2ms)
  ├── LIN 控制报文处理
  ├── 诊断会话超时检查
  ├── Flash Routine 0x27 例程处理
  └── 门驱动 PWM 控制
```

### 3.6 初始化顺序

```
TcMain()
  ├── DoorGpioInit()              ── PWM/GPIO 初始化
  ├── WdgInit()                   ── 看门狗 (可选)
  ├── TC_LOG_Init()               ── 调试日志

  │

```

## 4. 电容检测流程

电容检测基于 Tinychip 自容/互容检测方案，使用 `touch/` 中间件 + 预构建库 `silib.lib`。

### 4.1 架构与源文件

| 文件 | 作用 |
|------|------|
| `touch_config.c` | Sensor 引脚配置、阈值参数、扫描序列 |
| `si_touch_port.c` | TouchTask 任务回调、快慢扫切换、触摸数据上报 |
| `touch_haldispatch.c` | HAL 调度器: 状态机、扫描队列管理 |
| `touch_halnode.c` | 节点控制: 电荷转移、充放电时序 |
| `touch_tool.c` | 数字滤波、基线跟踪、环境补偿、触摸判决 |
| `silib.lib` (预构建) | Silabs 触摸算法库 |

### 4.2 检测流程

```
TouchTask 触发 (每 1ms)
      │
      ▼
touch_haldispatch 调度器
      │
      ├── 自容扫描 (Self-Capacitance)
      │     ├── 电极充电 → 测量充电时间/频率
      │     └── 原始数据采集 (由 silib 算法处理)
      │
      ├── 互容扫描 (Mutual-Capacitance)
      │     ├── TX 发送激励信号 → RX 测量耦合电容
      │     └── 原始数据采集
      │
      └── 数据处理 (touch_tool)
            ├── 数字滤波 (均值/低通)
            ├── 基线跟踪 (Baseline Tracking)
            ├── 环境补偿
            └── 触摸判决 (阈值比较 + 去抖动)
```

### 5.2 LIN 从节点状态机

```
        ┌──────────────┐
        │   UNINIT     │  ← 上电/复位初始状态
        └──────┬───────┘
               │ lin_lld_sci_init()
               ▼
        ┌──────────────┐
        │    IDLE      │  ← 等待总线活动
        └──────┬───────┘
               │ Break 检测 (LIN_INT_BREAK_DET_FLAG)
               ▼
        ┌──────────────┐
        │  RECV_SYN    │  ← 接收同步字节 (0x55)
        └──────┬───────┘
               │ Sync 确认
               ▼
        ┌──────────────┐
        │  RECV_PID    │  ← 接收保护标识符
        └──────┬───────┘
               │ PID 校验 → 匹配 NAD
               ▼
        ┌──────────────────┐
        │  RECV_DATA       │  ← 接收数据 (从机响应)
        │  or SEND_DATA    │  ← 发送数据 (从机发送)
        └──────────────────┘
               │ 校验和验证
               ▼
        ┌──────────────────┐
        │ PROC_CALLBACK    │  ← 触发 lin_process 回调
        └──────┬───────────┘
               │
               ▼
        ┌──────────────┐
        │    IDLE      │  ← 回到空闲
        └──────────────┘

        ┌──────────────────┐
        │ LIN_SLEEP_MODE   │  ← 总线空闲超时进入休眠
        └──────────────────┘
```

### 5.3 LIN 接收流程 (中断驱动)

```
LIN Break 到达
    │
    ▼
SCI 中断: BREAK_DET
    │  → state = RECV_SYN
    │  → 启动帧超时计数器
    │
    ▼
SCI 中断: SYNC_DET
    │  → 自动波特率校准
    │  → state = RECV_PID
    │
    ▼
SCI 中断: PID_DONE
    │  → 读取 PID, 校验奇偶性
    │  → 匹配成功 → 设置响应超时
    │
    ▼
SCI 中断: 1BYTE (逐字节)
    │  → 逐字节填充 response_buffer
    │  → 字节数 = 报文长度时:
    │      ├─ 校验和正确 → CALLBACK_RX_COMPLETED
    │      └─ 校验和错误 → CALLBACK_CHECKSUM_ERR
    │
    ▼
lin_process 回调 → LIN 协议层解析 → 诊断分发
```

### 5.4 LIN 发送流程

```
上层调用 ld_send_message() / lin_uds_send()
    │
    ▼
准备响应数据 (SID+0x40 正响应 或 0x7F+NRC)
    │
    ▼
数据写入 response_buffer
    │
    ▼
Lin Master 发送匹配 NAD 的 PID
    │
    ▼
SCI 中断: PID OK → 状态 = SEND_DATA
    │  → pal_lin_tx_response(pid, buf, len)
    │
    ▼
SCI 中断: TX_DONE
    │  → CALLBACK_TX_COMPLETED
    │  → state = IDLE
    │
    ▼
发送完成
```

## 6. UDS 诊断服务分发架构

### 6.1 双层分发机制

App 侧 UDS 服务采用**双层分发**架构:

```
LIN 传输层收到诊断请求
        │
        ▼
┌──────────────────────────────────────────┐
│ Layer 1: LIN 协议分发                     │
│ diagnosticIII.c / lin_diag_service_handle()│
│                                          │
│ 遍历 lin_diag_services_supported[] 匹配 SID│
│ 匹配成功后 SWITCH:                        │
│   0xB2 → lin_diagservice_read_by_id()     │
│   0xB6 → lin_diag_save_configuration()    │
│   0xB0 → lin_diagservice_assign_nad()     │
│   0xB7 → lin_diag_assign_frame_id_range()  │
│   0xB3 → lin_diag_conditional_change_nad() │
│   0xB1 → lin_diag_assign_frame_id()        │
│   0x2F → lin_diag_io_control_by_id()       │
│   0xB4 → 设置 g_bUDSDataDumpFlag          │
│   0x32 → lin_diag_get_traceability_msg()   │
│   0xB5 → lin_diag_target_reset()           │
│   0xA0 → 设置 g_bUDSReadLogInfo           │
│   0xAD/0xAE/0xAF → 一致性测试             │
│   default → ★ 下沉至 Layer 2 ★            │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│ Layer 2: 客户 UDS 分发                     │
│ custom_diagnosticIII.c                    │
│                                          │
│ lin_custom_diag_service_handle() (弱覆盖)  │
│   → 复制请求缓存                           │
│   → 调用 lin_handle_uds()                │
│                                          │
│ SWITCH on SID:                            │
│   0x10 → uds_diagnostic_session_control() │
│   0x11 → uds_diagnostic_rest()           │
│   0x14 → uds_diagnostic_clear_dtc_info() │
│   0x22 → uds_diagnostic_readdata_by_id() │
│   0x28 → uds_communction_control()       │
│   0x2E → NRC 0x33 (拒绝写入)             │
│   0x31 → uds_diagnostic_route_control()  │
│   0x3E → uds_tester_present_control()    │
│   0x85 → uds_diagnostic_dtc_control()    │
│   0xB5 → uds_diagnostic_assign_NAD()     │
│   0x27/0x34/0x36/0x37 → NRC 0x7F        │
└──────────────────────────────────────────┘
```

### 6.2 Layer 1 已注册服务表

`lin_diag_services_supported[]` (定义于 `lin_cfg.c`) 共 **27 个 SID**:

```c
const l_u8 lin_diag_services_supported[27] = {
    0x10, 0x11, 0x14, 0x19, 0x22, 0x27, 0x28,
    0x2E, 0x2F, 0x31, 0x3E, 0x34, 0x36, 0x37,
    0x85, 0xA0,
    0xB0, 0xB1, 0xB2, 0xB3, 0xB4, 0xB5, 0xB6, 0xB7,
    0xBA, 0xBB, 0xBC
};
```

所有注册 SID 经由 `lin_diag_services_flag[]` 标志位标记请求到达。Layer 1 通过 switch/case 仅处理 LIN 标准配置服务，其余下沉至 Layer 2。

### 6.3 关键依赖关系

- **`sid_0x*.c` 文件**: 为 Layer 1 (LIN 协议服务) 提供独立处理器函数，14 个编译文件对应 14 个 SID
- **`custom_diagnosticIII.c`**: 为 Layer 2 (UDS 应用服务) 提供所有处理器函数
- **`diagnosticIII.c`**: 分发枢纽，调用 `lin_custom_diag_service_handle()` (弱符号默认空实现)
- **`linlib_seres.lib`**: 预构建 LIN 协议栈库，包含传输层、会话层管理、部分标准服务处理器

## 7. 正常工作模式 / 升级模式切换逻辑

### 7.1 模式定义

| 运行模式 | 执行固件 | 描述 |
|---------|---------|------|
| **正常运行模式** | App | 电容检测、LIN 通信、UDS 诊断、门控制 |
| **Bootloader 升级模式** | Boot | 等待 UDS 编程会话，擦写 Flash，固件下载 |
| **过渡态 (IDLE 等待)** | Boot | 等待约 50ms 后基于 DFU 信息决策 |

### 7.2 正常 → 升级切换

```
正常运行模式 (App)
    │
    ├── App 收到 $10 $02 (编程会话)
    │     └── 触发 NVIC_SystemReset()
    │
    ▼
系统复位 → Bootloader 启动
    │
    ├── 初始化 LIN, 进入 main_loops()
    │
    ├── [IDLE 约 50ms 窗口内]
    │     └── 收到 $10 $02?
    │           ├── YES → BOOT_STATE_UPGRADE → OTA 流程
    │           └── NO  → 检查 DFU 信息
    │                       ├── 有效 → BOOT_STATE_USER_APP → JumpToApp()
    │                       └── 无效 → BOOT_STATE_UPGRADE → 等待 OTA
    │
    └── 42ms 后: DFU 信息有效 → JumpToApp()
```

### 7.3 升级 → 正常切换

```
Bootloader 升级模式
    │
    ├── UDS OTA 流程完成 (CRC 校验通过)
    │     └── 写入 DFU 信息 (magic=0xDEADBEEF, reason=SUCCESS)
    │
    ├── $31 $01 $7221 (复位) → NVIC_SystemReset()
    │
    ▼
Bootloader 重启 → DFU 信息有效 → JumpToApp() → 新固件
```

### 7.4 升级失败处理

```
OTA 过程超时 (3 秒无 UDS 请求)
    │
    ▼
dfu_process_exit(DFU_MSG_TIMEOUT)
    │
    ├── 清除上下文
    ├── boot_state = BOOT_STATE_UPGRADE
    └── 等待下一次 OTA

Flash 擦写失败
    │
    ▼
dfu_process_exit(DFU_MSG_ERASE_ERROR)
    │
    └── 不写入 DFU 信息，下次上电仍进入 Boot
```

## 8. 关键数据结构

### 8.1 DFU 上下文 (Boot 侧)

```c
typedef struct {
    uint32_t write_addr;             // 当前写入 Flash 地址
    uint32_t write_length;           // 已写入字节数
    uint32_t write_index;            // 写入数据包序号
    uint32_t write_crc;              // 实时 CRC32 校验值
    uint32_t receive_length;         // 已接收字节数
    uint32_t recevice_index;         // 已接收数据包序号
    uint32_t uds_timeout;            // UDS 超时计数 (ms)
    uint8_t  op_code;                // 当前操作码 (DFU_CMD_xxx)
    uint8_t  resp_value;             // 响应值/错误码
    uint8_t  program_flag;           // 编程标志
    uint8_t  boot_state;             // Boot 状态 (IDLE/USER_APP/UPGRADE)
    lin_baudrate_config_t lin_config;      // LIN 波特率配置
    last_dfu_info_t dfu_info;              // 上次 DFU 信息
    queue_list_t queue_list;               // 数据包双缓冲队列
} dfu_manager_context_t;
```

### 8.2 DFU 信息结构

```c
typedef struct {
    uint32_t magic;                  // 魔数 0xDEADBEEF (有效性标记)
    uint32_t image_size;             // 固件镜像大小
    uint32_t image_crc;              // 固件 CRC32 校验值
    uint32_t written_image_length;   // 已写入长度 (未使用)
    uint32_t written_image_crc;      // 已写入 CRC (未使用)
    uint32_t reason;                 // 升级结果 (SUCCESS/ERROR)
    uint32_t version;                // 固件版本号
    uint32_t image_addr;             // 固件存储起始地址
    uint32_t time[3];                // 编译时间戳
} last_dfu_info_t;
```
